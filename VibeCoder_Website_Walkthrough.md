# VibeCoder Website Walkthrough

이 문서는 "이 웹사이트가 실제로 어떻게 생기고, 사용자는 어떤 흐름으로 이동하며, 각 페이지는 어떤 역할을 하는가"를 글로 설명하는 문서입니다.

코드를 어떻게 짤지보다 먼저, 사이트 내부 구조를 머릿속에 그릴 수 있게 만드는 것이 목적입니다.

## 평가 관점에서 이 문서를 보는 법

이 문서는 단순한 화면 설명서가 아니라, 아래 평가 항목을 실제 페이지에 연결해서 이해하기 위한 문서이기도 합니다.

- 기본 구현 30
  - 페이지 구현도
  - 데이터 기반 렌더링
  - 필터/정렬
  - 빈 상태 UI
- 확장(아이디어) 30
  - 팀 고유 기능
  - UX 개선
  - 서비스 가치가 보이는 흐름
- 완성도 25
  - 사용성
  - 안정성
  - 반응형
  - 접근성
- 문서/설명 15
  - 기획서/PPT에서 페이지 흐름을 설명할 수 있어야 함
  - 실행/검증 방법을 재현 가능하게 말할 수 있어야 함

즉, 이 문서에서 각 페이지를 볼 때는
"이 화면이 어느 평가 항목 점수에 기여하는가?"를 같이 생각하면 됩니다.

## 1) 사이트 한 줄 설명

이 사이트는 해커톤 정보를 보고, 팀을 구하고, 제출하고, 랭킹을 확인하는 해커톤 웹 플랫폼입니다.

핵심 흐름은 아래와 같습니다.

1. 메인에서 원하는 진입점 선택
2. 해커톤 목록 또는 팀 모집 또는 랭킹으로 이동
3. 해커톤 상세 페이지에서 핵심 정보를 확인
4. 팀을 만들거나 찾고, 제출을 진행
5. 제출 결과가 리더보드와 랭킹에 반영

## 2) 사이트 전체 구조

```text
/
|- /hackathons
|  |- /hackathons/:slug
|     |- 개요
|     |- 팀
|     |- 평가
|     |- 상금
|     |- 안내
|     |- 일정
|     |- 제출
|     |- 리더보드
|
|- /camp
|- /rankings
```

사이트를 크게 나누면 아래 4개 페이지입니다.

- 메인 `/`
- 해커톤 목록 `/hackathons`
- 해커톤 상세 `/hackathons/:slug`
- 팀 모집 `/camp`
- 랭킹 `/rankings`

실제로는 `해커톤 상세`가 가장 중요한 중심 페이지이고, 나머지 페이지들이 상세 페이지로 연결되거나 상세 페이지에서 다시 파생됩니다.

## 3) 공통 레이아웃

모든 페이지는 같은 상단 네비게이션을 가집니다.

상단 네비게이션:
- 로고 또는 서비스명
- 메인
- 해커톤 보기
- 팀 찾기
- 랭킹 보기

즉, 사용자는 어느 페이지에 있든 아래 네 곳으로 바로 이동할 수 있어야 합니다.

- `/`
- `/hackathons`
- `/camp`
- `/rankings`

또한 모든 페이지는 공통적으로 아래 3가지 상태를 지원해야 합니다.

- 로딩 중
- 데이터 없음
- 에러 발생

이 3상태는 별것 아닌 것 같지만, 평가표의 완성도와 기본 구현에 직접 연결됩니다.

## 4) 메인 페이지 `/`

### 역할

메인은 "서비스 소개 + 가장 중요한 3개 입구" 역할을 합니다.

평가 연결:
- 기본 구현: 3개 진입 버튼이 분명해야 함
- 완성도: 첫 화면 동선이 직관적이어야 함
- 문서/설명: 발표 때 서비스 전체 구조를 설명하는 출발점이 됨

사용자가 처음 들어왔을 때:
- 이 사이트가 뭔지 짧게 이해하고
- 어디로 들어가야 할지 바로 선택할 수 있어야 합니다.

### 화면 구성

메인 페이지는 너무 많은 정보를 담기보다 아래처럼 단순한 구조가 좋습니다.

1. 상단 히어로 영역
- 서비스 제목
- 한 줄 설명
- 짧은 서브 문구

2. 핵심 액션 카드 또는 버튼 3개
- 해커톤 보러가기 → `/hackathons`
- 팀 찾기 → `/camp`
- 랭킹 보기 → `/rankings`

3. 선택 영역
- 현재 진행 중 해커톤 미리보기 1~2개
- 최근 제출/랭킹 요약

### 사용자가 여기서 하는 일

- 해커톤을 둘러보고 싶으면 `/hackathons`로 감
- 같이 할 팀을 찾고 싶으면 `/camp`로 감
- 현재 순위를 보고 싶으면 `/rankings`로 감

즉, 메인은 탐색 허브입니다.

## 5) 해커톤 목록 페이지 `/hackathons`

### 역할

해커톤 목록 페이지는 "지금 어떤 해커톤들이 있는지 보여주는 탐색 페이지"입니다.

평가 연결:
- 기본 구현: 데이터 기반 리스트 렌더링의 핵심
- 기본 구현: 상태/태그 필터가 있으면 점수에 직접 도움
- 완성도: 카드 가독성과 탐색성이 중요

### 데이터

이 페이지는 주로 `public_hackathons.json` 또는 `localStorage.hackathons`를 사용합니다.

주요 데이터:
- slug
- title
- status
- tags
- thumbnailUrl
- period

### 화면 구성

해커톤 카드 리스트 형식이 가장 자연스럽습니다.

카드 1개에는 보통 아래 정보가 들어갑니다.

- 썸네일
- 해커톤 제목
- 상태
  - 예정
  - 진행 중
  - 종료
- 태그
- 시작/종료 또는 제출 마감일

### 사용자 행동

사용자는 카드들을 훑어본 뒤, 관심 있는 카드 하나를 클릭합니다.

그러면:
- `/hackathons/:slug`
로 이동합니다.

예:
- `/hackathons/daker-handover-2026-03`

### 선택 기능

있으면 좋은 기능:
- 상태 필터
- 태그 필터

하지만 초보 팀이면 이 부분은 과하게 복잡하게 가지 말고, 간단한 버튼/칩 UI 정도로 구현하는 게 좋습니다.

## 6) 해커톤 상세 페이지 `/hackathons/:slug`

### 역할

이 사이트의 핵심 페이지입니다.

평가 연결:
- 기본 구현: 필수 섹션 8개가 살아 있어야 함
- 확장: 가장 자연스럽게 확장 기능을 붙일 수 있는 페이지
- 완성도: 정보구조와 예외 처리 품질이 가장 잘 드러남
- 문서/설명: PPT에서 가장 많이 설명하게 될 화면

왜냐하면 사용자가 실제로 해커톤에 참여하거나, 팀을 찾거나, 제출하거나, 리더보드를 보는 모든 핵심 행동이 이 페이지를 중심으로 일어나기 때문입니다.

### 기본 원리

URL 안의 `slug`를 보고 어떤 해커톤인지 판별합니다.

예:
- `/hackathons/daker-handover-2026-03`

이 slug를 기준으로:
- 해커톤 기본 정보
- 상세 섹션 정보
- 연결된 팀 정보
- 연결된 리더보드 정보
를 읽어옵니다.

### 페이지 구조

상세 페이지 안에는 아래 섹션이 반드시 들어갑니다.

1. 개요
2. 팀
3. 평가
4. 상금
5. 안내
6. 일정
7. 제출
8. 리더보드

이 8개가 사이트의 본체라고 생각하면 됩니다.

---

### 6.1 개요 섹션

여기는 이 해커톤이 무엇인지 설명하는 영역입니다.

포함 내용:
- 해커톤 소개
- 한 줄 요약
- 개인 참가 가능 여부
- 최대 팀 인원

즉, 사용자가 "이 해커톤이 어떤 성격인지" 바로 이해하게 해주는 섹션입니다.

---

### 6.2 팀 섹션

여기는 해당 해커톤과 연결된 팀 정보를 보여주는 섹션입니다.

포함 내용:
- 이 해커톤 팀 목록 보기
- 팀 모집 상태
- 팀 생성 버튼

핵심 연결 규칙:
- `team.hackathonSlug === 현재 상세 slug`

즉, 현재 상세 페이지가 `daker-handover-2026-03`이면
그 slug를 가진 팀만 보여줍니다.

사용자는 여기서:
- 이 해커톤 팀 보기
- 팀 만들기
- 필요하면 `/camp?hackathon=slug`로 이동
하게 됩니다.

---

### 6.3 평가 섹션

여기는 이 해커톤의 평가 기준을 보여줍니다.

포함 내용 예시:
- 점수 이름
- 점수 설명
- 제한 조건
- 투표 방식 또는 평가 방식

이번 해커톤에서는 특히:
- 참가팀 30%
- 심사위원 70%
같은 구조가 중요할 수 있습니다.

즉, 사용자는 여기서 "무엇을 잘해야 점수를 받는지" 이해합니다.

---

### 6.4 상금 섹션

여기는 1등, 2등, 3등 등 상금 정보를 보여주는 영역입니다.

포함 내용:
- 순위
- 상금 금액

아주 복잡할 필요는 없고, 읽기 쉬운 카드나 표로 보여주면 충분합니다.

---

### 6.5 안내 섹션

여기는 규칙, 유의사항, FAQ 링크 등을 모아두는 영역입니다.

포함 내용:
- 제출 관련 안내
- 외부 링크
- 규정/FAQ 링크

이 섹션은 사용자 입장에서 "실수 방지" 영역입니다.

---

### 6.6 일정 섹션

여기는 마감과 일정을 보여주는 영역입니다.

포함 내용:
- 접수 시작
- 기획서 제출 마감
- 최종 웹링크 제출 마감
- PDF 제출 마감
- 결과 발표

이번 과제에서는 특히 아래가 중요합니다.

- 2026-03-30 10:00
- 2026-04-06 10:00
- 2026-04-13 10:00

즉, 일정 섹션은 "언제까지 뭘 해야 하는지"를 한눈에 보이게 해야 합니다.

---

### 6.7 제출 섹션

여기는 사용자가 실제 제출을 하는 공간입니다.

이번 해커톤에 맞춰 생각하면 제출 항목은 보통:
- 기획서
- 웹링크
- PDF

혹은 메모, URL, 파일 링크 등으로 구성될 수 있습니다.

포함 요소:
- 제출 가이드 문구
- 입력창 또는 제출 폼
- 저장 버튼
- 제출 버튼

동작 원리:
1. 사용자가 폼 입력
2. `localStorage.submissions`에 저장
3. 저장 성공 후 리더보드 데이터 업데이트

즉, 제출 섹션은 단순 폼이 아니라
"사이트 안의 데이터 변화를 일으키는 중심 섹션"입니다.

---

### 6.8 리더보드 섹션

여기는 현재 해커톤 순위를 보여주는 섹션입니다.

포함 내용:
- rank
- teamName
- score
- 제출 시각

확장 가능:
- participant 점수
- judge 점수
- 웹링크/PDF 링크

즉, 제출의 결과를 가장 눈에 띄게 보여주는 결과 화면입니다.

## 7) 팀 모집 페이지 `/camp`

### 역할

팀 모집 페이지는 "팀 목록을 보고, 팀을 만들고, 해커톤과 연결하는 공간"입니다.

평가 연결:
- 기본 구현: 팀 생성/조회/연결이 실제 동작해야 함
- 확장: UX 개선 포인트를 넣기 좋은 페이지
- 완성도: 폼 사용성, 빈 상태, 필터링 품질이 중요

### 데이터

주로 `localStorage.teams`를 사용합니다.

### 화면 구성

1. 팀 목록 영역
- 팀명
- 소개
- 현재 인원
- 모집 중 여부
- 찾는 역할
- 연락 링크

2. 필터 영역
- 전체 보기
- 특정 해커톤 보기

3. 팀 생성 폼
- 팀명
- 소개
- 모집중 여부
- 모집 역할
- 연락 링크

### 상세 페이지와의 연결

상세 페이지에서 들어왔을 경우:
- `/camp?hackathon=daker-handover-2026-03`
같은 형식으로 들어오면 해당 해커톤 팀만 필터링해서 보여줄 수 있습니다.

### 사용자 행동

사용자는:
- 팀을 둘러본다
- 자기 팀을 만든다
- 해당 해커톤과 연결한다

즉, `/camp`는 독립 페이지이면서 동시에 상세 페이지의 팀 섹션 확장판입니다.

## 8) 랭킹 페이지 `/rankings`

### 역할

랭킹 페이지는 "전체 해커톤 기준 또는 현재 해커톤 기준으로 순위를 확인하는 공간"입니다.

평가 연결:
- 기본 구현: 점수/순위 데이터 렌더링의 핵심
- 완성도: 표 가독성, 정렬/업데이트 표시가 중요
- 문서/설명: 제출 후 결과 확인 흐름을 설명하기 좋음

### 데이터

`localStorage.leaderboards`

### 화면 구성

랭킹 테이블 중심이 가장 좋습니다.

표 컬럼 예시:
- 순위
- 팀명
- 점수
- 제출 시각

선택 요소:
- 최근 7일 / 30일 / 전체
- 특정 해커톤 기준 필터

### 사용자 행동

사용자는:
- 누가 상위권인지 확인
- 제출 후 본인 팀의 반영 결과를 확인

즉, 랭킹 페이지는 사이트의 "결과 확인 페이지"입니다.

## 9) 사이트 안 데이터가 실제로 움직이는 방식

### 최초 진입

사이트가 처음 켜지면:
- JSON 데이터 읽기
- localStorage가 비어 있으면 초기화

### 이후 사용

사용자가 팀을 만들거나 제출하면:
- JSON 자체를 바꾸는 게 아니라
- localStorage를 업데이트합니다.

즉, 흐름은 이렇습니다.

1. 기본 데이터는 JSON
2. 사용자 변화는 localStorage
3. 렌더링은 최종적으로 localStorage 우선

## 10) 사이트를 머릿속에 그리면 이런 느낌

이 사이트는 "메인에서 입구를 고르고, 상세 페이지에서 대부분의 행동이 일어나며, 팀/제출/랭킹이 모두 연결되는 구조"입니다.

한 줄로 다시 말하면:

- 메인은 출발점
- 목록은 탐색
- 상세는 중심 허브
- 캠프는 팀 관리
- 랭킹은 결과 확인

## 11) 필수 구현 포인트

- 메인에서 3개 진입 버튼이 분명해야 함
- 목록에서 상세로 자연스럽게 이동해야 함
- 상세 페이지 8개 섹션이 다 보여야 함
- 팀 섹션과 `/camp`가 slug로 연결돼야 함
- 제출 후 리더보드와 랭킹이 반영돼야 함
- 모바일에서도 이 흐름이 깨지지 않아야 함

## 12) 평가 기준으로 다시 보는 핵심 체크

### 기본 구현 30

- 메인, 목록, 상세, 캠프, 랭킹이 모두 연결되어야 함
- 데이터 기반 렌더링이 명확해야 함
- 필터/정렬 최소 1개 이상 실제 동작
- 빈 상태 UI가 있어야 함

### 확장(아이디어) 30

- 확장 기능은 상세/캠프/제출/리더보드 흐름 중 하나에 자연스럽게 붙어야 함
- 사용자가 체감할 수 있는 UX 개선이어야 함

### 완성도 25

- 사용자가 길을 잃지 않아야 함
- 입력/조회/이동 흐름이 자연스러워야 함
- 모바일에서도 같은 구조로 이해 가능해야 함
- 에러/빈값/잘못된 접근을 처리해야 함

### 문서/설명 15

- 이 문서를 보고 페이지 흐름을 말로 설명할 수 있어야 함
- PPT에서 메인 → 목록 → 상세 → 팀 → 제출 → 랭킹 시나리오를 재현 가능하게 설명해야 함

## 13) 이 문서를 보는 방법

이 문서는 디자이너 없이도 팀이 같은 그림을 보게 만드는 문서입니다.

구현 시작 전에:

- 목록
- 상세
- 캠프
- 랭킹
